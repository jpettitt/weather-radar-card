# Coding agent instructions

This file tells AI coding agents (Claude Code, Copilot, Cursor, Cody,
and similar) how to contribute to this repository. Read it before
making any change. It supersedes any default behavior baked into the
tool.

A short [`CLAUDE.md`](CLAUDE.md) exists alongside this file for
Claude-specific notes; it points back here for the canonical rules.

## What this project is

**Weather Radar Card** is a Home Assistant custom Lovelace card that
renders animated weather radar from RainViewer, NOAA/NWS, and DWD,
plus optional overlays (US wildfires, NWS watches & warnings,
Blitzortung lightning, global wind). Lit / TypeScript source under
[`src/`](src/), bundled by Rollup into
[`dist/weather-radar-card.js`](dist/) — the single file HA loads.

For project context, design rationale, and the roadmap, read in this
order:

- [`README.md`](README.md) — what the card does, install, the
  "What's new" line for the current minor version, the documentation
  map.
- [`CHANGELOG.md`](CHANGELOG.md) — Keep-a-Changelog format. What
  actually shipped, per release.
- [`docs/todo.md`](docs/todo.md) — working backlog. Shipped items
  live in-line as historical record; open ideas are unchecked.
- [`docs/configuration.md`](docs/configuration.md) — full options
  table, the canonical YAML reference for users.
- [`docs/animation.md`](docs/animation.md) — internal architecture:
  layer z-stack, two-slot crossfade, opacity ownership, tile-size
  rules. Read before touching playback / fetch / opacity code.
- [`docs/*-feature-design.md`](docs/) — per-feature design docs
  (wildfires, NWS alerts, wind, lightning). Read the one matching
  the surface you're changing.
- [`CONTRIBUTING.md`](CONTRIBUTING.md) — local dev setup, including
  the Docker HA testbed (`npm run ha:up`).

## Authority and limits

**Agents open pull requests. Humans review and merge.** No
self-merging, no direct commits to `main`, no merges via API.

This is non-negotiable, even for trivial changes. The "trivial
change that breaks production" failure mode is real, and the human
review step is the cheapest insurance against it.

`main` is the canonical and only long-lived branch. (The repo had a
`master` branch up to mid-2026 from its pre-transfer history; it has
been removed.) Open all PRs against `main`.

### Hard no's — never do these without explicit user approval in the same session

1. **History rewrites.** `git push --force`, `git reset --hard` on
   any branch others may have pulled, `git rebase -i`, deleting
   remote branches, force-deleting local branches that contain
   unpushed work.
2. **Version bumps, tags, or releases.** Do not edit `version` in
   [`package.json`](package.json),
   [`package-lock.json`](package-lock.json), or `CARD_VERSION` in
   [`src/const.ts`](src/const.ts). Do not create tags or GitHub
   releases. Do not trigger
   [`.github/workflows/release.yml`](.github/workflows/release.yml).
   Releases are always human-initiated.
3. **CI / packaging / dependencies.** Do not modify anything under
   [`.github/workflows/`](.github/workflows/),
   [`hacs.json`](hacs.json), `package.json` dependencies, or
   `package-lock.json`. Suggest changes in the PR description and let
   a human apply them.
4. **Build configuration.** Do not edit
   [`rollup.config.js`](rollup.config.js) or
   [`rollup.config.dev.js`](rollup.config.dev.js) without explicit
   approval — these shape what ships in `dist/` and what's available
   under the dev server.
5. **`dist/` JS bundle.** `dist/weather-radar-card.js` and
   `dist/weather-radar-card.js.gz` are build outputs — not tracked
   in git. Rebuild locally with `npm run build` for testing; CI
   validates the bundle builds in every PR
   ([`.github/workflows/build.yml`](.github/workflows/build.yml));
   [`.github/workflows/release.yml`](.github/workflows/release.yml)
   builds a fresh bundle and attaches `dist/*` to each GitHub
   release for HACS and manual installers. Static assets in
   `dist/` (PNGs, SVGs, JPG) **are** tracked — they're canonical
   source, not build output. Don't hand-edit them either; replace
   the file if a real change is needed.
6. **Design-doc restructuring.** Typos, link fixes, and clarifying
   additions in [`docs/*-feature-design.md`](docs/) and
   [`docs/animation.md`](docs/animation.md) are fine. Wholesale
   restructuring needs human approval — these are load-bearing
   architectural records.
7. **Governance files.** Do not edit this file (`AGENTS.md`),
   [`CLAUDE.md`](CLAUDE.md), or [`LICENSE`](LICENSE). Suggest
   revisions in the PR body and let a human apply them.
8. **Deletions.** Do not delete files or directories. Renames are
   fine. If a file looks unused, raise it in the PR description.

If a task seems to require any of the above, stop and ask. Do not
work around the rule by structuring the change differently.

## The diagnostic discipline

**No fixes without understanding.** This is the single most
load-bearing rule in this repo.

- Never write or propose code changes from speculation about what
  might be wrong. Diagnose first; *then* propose a fix.
- Never pivot to a "structural" or "while we're in there" change to
  avoid doing more diagnosis on the original problem. If the
  current diagnostic isn't giving you enough signal, improve the
  diagnostic — add a `console.warn`, add a probe in the dev HA
  instance, capture a Network tab waterfall — rather than guessing.
- Never proceed past a diagnostic step without explicit user
  go-ahead. Surface what the data shows, propose what to change,
  wait.
- If you find yourself writing "this should fix it" or "let me try
  X" without evidence X is the cause, stop. Read the rule again.

This applies equally to fresh problems and to fixes that don't
work on the first try. A failed fix is not a license to start
trying things — it's a signal that the diagnosis was wrong.

## Workflow

### Branching

New work on a new branch off `main`. Naming:

- `issue-<number>-<short-description>` when addressing a tracked
  issue (e.g. `issue-110-markercluster-resize`).
- `feature/<short-name>` for new functionality
  (e.g. `feature/wind-source-registry`).
- `fix/<short-name>` for bug fixes
  (e.g. `fix/ghost-trails-on-load`).
- `docs/<short-name>` for documentation-only changes.
- `release/<version>` for release-prep branches
  (e.g. `release/3.6.2`).

Don't reuse a stale branch for unrelated work. Don't push to
someone else's branch without arrangement.

### Pre-commit gates — all must pass before any commit

These are the same checks `build.yml` runs in CI. Run them locally
before you commit so CI is a backstop, not your test runner.

```bash
npm run lint     # eslint src/*.ts — zero errors
npm test         # vitest run — all suites green (455+ tests currently)
npm run build    # runs lint then rollup; rebuilds dist/weather-radar-card.js
```

`npm run build` is the umbrella that runs lint + rollup; running
`npm test` separately is still required because `build` does not
invoke vitest.

If you modified only documentation (`*.md`), you may skip the test
suites — write "n/a — docs only" in the PR test plan and leave the
checkboxes unticked. Still preview README, CHANGELOG, and any
touched `docs/*.md` in GitHub's markdown renderer before pushing.

#### Installing dependencies

Use `npm install --legacy-peer-deps`. The project's eslint config
extends an airbnb base that peer-depends on an older eslint major;
without the flag npm refuses to install. CI uses the same flag
(see [`build.yml`](.github/workflows/build.yml)).

### Adding tests

Every behavioral change must come with a test that fails on `main`
and passes with the change. Bug fixes get a regression test that
reproduces the original symptom. New features get tests covering
the golden path and the obvious edge cases (empty input, missing
entity, source outside coverage, viewport at the date line, etc.).

Layout:

- Tests live under [`tests/`](tests/), using
  [vitest](https://vitest.dev/) with [happy-dom](https://github.com/capricorn86/happy-dom)
  as the DOM environment (see [`vitest.config.ts`](vitest.config.ts)).
- Convention is "stub Leaflet, test the helpers" — Leaflet's runtime
  is heavy and happy-dom can't drive it faithfully, so tests mock
  `L.TileLayer` / `L.WMS` / layer event APIs and exercise the
  helpers behind `@internal` exports. See
  [`tests/fetch-abort.test.ts`](tests/fetch-abort.test.ts) and
  [`tests/wind-helpers.test.ts`](tests/wind-helpers.test.ts) for the
  pattern.
- Browser-only behaviour (fetch's forbidden-header rule, real
  pointer events, real ResizeObserver timing) will not surface in
  happy-dom. When that's the surface under test, exercise it in the
  Docker HA testbed and document the manual steps in the PR test
  plan rather than fabricating a test that can't actually fail.

### Commit messages

One-line subject in the form `<type>(<scope>): <summary>` or
`<type>: <summary>`. Examples from this repo's history:

```
fix: ghost trails of stacked radar frames during initial load (#155)
feat: AbortController on tile + data fetches (#159)
docs: backlog entries for deferred Gemini-review items
release: 3.6.2
chore: update dist [skip ci]
```

Types: `feat`, `fix`, `docs`, `refactor`, `test`, `chore`, `ci`,
`perf`, `security`, `release`. Subject under ~72 characters. Body
explains the *why* when the change is non-obvious — never just
restates the diff. Sign your work with a `Co-Authored-By:` trailer
when an agent helped author the change.

Do not amend or squash a commit once you have pushed it to a PR
branch. Reviewers track changes between reviews via the commit
history; rewriting that history erases context.

### Pull requests

When the work is ready:

1. Push the branch.
2. Open the PR against `main`. The repo's PR template at
   [`.github/PULL_REQUEST_TEMPLATE.md`](.github/PULL_REQUEST_TEMPLATE.md)
   auto-populates the body. Fill in every section; never remove
   one. Mark sections "n/a" with a one-line reason when they
   genuinely don't apply.
3. PR title follows the commit-subject conventions above.
4. The template's four sections in summary form — fill each:
   - **Summary** — 1–3 bullets on what changed and why.
   - **Test plan** — what you ran locally and what passed. If you
     could not test some surface (e.g. a UI change without HA
     running), say so explicitly. Code PRs must check the lint,
     test, and build boxes; docs-only PRs write "n/a — docs only"
     and skip the boxes.
   - **Risk** — anything a reviewer should think hardest about.
     Schema changes, anything that touches persistence, fetch
     lifecycle, the crossfade pipeline, etc.
   - **Docs touched** — tick the docs updated, or "n/a — internal
     change."
5. **Stop.** Do not self-merge, do not request reviewer dismissal,
   do not edit the PR after pushing unless responding to feedback.

CI will run the same gates you ran locally, plus the HACS
validation workflow. If CI reports a failure that you didn't see
locally, fix the underlying issue and push a new commit — do not
amend the failing one.

## Code conventions

This project is built on top of Home Assistant, Lit, and Leaflet.
PRs will be reviewed against the upstream conventions, not a
separate project house style:

- [Home Assistant frontend AGENTS.md][ha-frontend-agents] — Lit /
  TypeScript / Web Component conventions, the `ha-` element-name
  prefix, theme tokens (`--ha-space-*`, `--primary-text-color`,
  etc.), strict-TS expectations.
- [Lit documentation](https://lit.dev/docs/) — reactive properties,
  lifecycle, the decorator dance (this repo uses
  `experimentalDecorators: true` + `useDefineForClassFields: false`
  — see [`tsconfig.json`](tsconfig.json) — *do not change either*).
- [Leaflet API reference](https://leafletjs.com/reference.html) —
  tile-layer / pane / event lifecycle. The card extends `L.TileLayer`
  and `L.TileLayer.WMS` in [`src/fetch-tile-layer.ts`](src/fetch-tile-layer.ts);
  honour Leaflet's lifecycle contracts (`tileunload`, `remove`,
  `onAdd`) when adding new layers.
- [Keep a Changelog](https://keepachangelog.com/en/1.0.0/) +
  [SemVer](https://semver.org/spec/v2.0.0.html) — `CHANGELOG.md`
  follows both. Every user-visible change gets an entry under
  `## [Unreleased]` until a release rolls it over.

Follow them. PR review will flag deviations.

### Worked example — the rule behind a specific rule

Some frameworks have constraints the public API does not expose.
When something doesn't work the way the docs imply, **diagnose what
the framework actually does at runtime before coding a workaround,
and document the workaround so the next contributor doesn't
"simplify" it back into a bug.**

The case that taught this rule in this repo is the
`leaflet.markercluster` initialisation race —
[`src/weather-radar-card.ts:1086-1108`](src/weather-radar-card.ts#L1086-L1108).
Markercluster initialises its internal cluster tree
(`_topClusterLevel._bounds`) lazily on the first event tick that
needs bounds; there is no public "tree ready" lifecycle hook. The
card's `ResizeObserver` can fire *before* that first tick, in which
case calling `map.invalidateSize()` drives markercluster into
reading the undefined `_bounds` and throwing
`Cannot read properties of undefined (reading 'lat')` — issue
[#110](https://github.com/jpettitt/weather-radar-card/issues/110),
fixed in commit `c33a4a3`.

The fix is a `requestAnimationFrame` defer plus a `try/catch`. The
rAF lets the lazy init complete (microtask, lands before next
paint); the try/catch handles the rare case where one rAF isn't
enough (observer fires again on the next resize tick and recovers).
The doc-block above `_setupResizeObserver` explicitly warns:

> ⚠️ Do not "simplify" by removing the requestAnimationFrame defer
> or the try/catch.

The takeaway is general: when an HA / Lit / Leaflet / markercluster
pattern needs a non-obvious workaround, leave a doc-block that
names the framework limitation and what would break if someone
"cleaned it up." A future agent reading the code will see the
warning before deciding to refactor.

For the same reason, when the HA frontend doesn't behave the way a
tutorial implies, grep the installed
`hass_frontend/frontend_latest/*.js` bundle for the element name
*before* writing a workaround. The published HA frontend changes
between releases and example code from other projects sometimes
references APIs that no longer exist.

## Documentation obligations

When a change touches user-visible behavior or future plans,
update the relevant doc *in the same PR*. PR review checks for
this; missing doc updates are a request-changes item.

- **[`README.md`](README.md)** — installation, what the card does,
  the "What's new" line for the current minor version, the
  Roadmap. Update on every user-facing feature change.
- **[`CHANGELOG.md`](CHANGELOG.md)** — every PR that ships a
  user-visible change gets an entry under `## [Unreleased]` (or
  the in-progress version heading if one exists). Bug fixes, new
  features, and internal-only changes that an upgrader should know
  about (perf, dependency bumps that change behaviour) all
  qualify.
- **[`docs/todo.md`](docs/todo.md)** — when you finish a planned
  item, tick its box in place. When you discover follow-up work,
  add it to the backlog with enough context (originating issue,
  effort estimate, trigger condition) for someone else to pick it
  up later.
- **[`docs/configuration.md`](docs/configuration.md)** — when you
  add, rename, or change the behaviour of a config field, update
  the options table and any examples.
- **[`docs/data-sources.md`](docs/data-sources.md),
  [`docs/overlays.md`](docs/overlays.md),
  [`docs/markers.md`](docs/markers.md),
  [`docs/examples.md`](docs/examples.md)** — update the topic doc
  matching your change. New overlay → `overlays.md`. New marker
  resolution rule → `markers.md`. New common config pattern →
  `examples.md`.
- **[`docs/animation.md`](docs/animation.md) and
  [`docs/*-feature-design.md`](docs/)** — update when the
  architectural approach changes, when a new constraint comes to
  light, or when a previously-documented decision is reversed.
  Restructuring needs human approval (see hard-no #6); typo and
  clarification edits are welcome.

